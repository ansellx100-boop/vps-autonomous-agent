# Сценарий: мониторинг производственной безопасности в горнодобыче

Задача: круглосуточный сбор данных о производственной безопасности в горнодобывающей отрасли → база данных → еженедельный PDF-отчёт в Telegram (или по запросу).

## Что нужно добавить к текущему агенту

Текущий агент только передаёт текст в LLM и пишет ответ в логи. Для этого сценария нужны отдельные модули.

| Компонент | Назначение | Варианты реализации |
|-----------|------------|----------------------|
| **Поиск в интернете** | Сбор новостей, отчётов, нормативов | **Реализовано:** DuckDuckGo через `duck-duck-scrape` (без API-ключа). Задача с `type: "collect"` или текст про «искать в интернете» + «производственная безопасность»/«горнодобыва» запускает поиск, результаты пишутся в `data/search-results.jsonl`. |
| **Планировщик 24/7** | Запуск сбора по расписанию | `node-cron` или внешний cron → вызов `/task` с типом `collect` |
| **База данных** | Хранение собранных материалов | SQLite (файл) или PostgreSQL (если есть на Railway); таблицы: источники, статьи, даты, теги |
| **Генерация PDF** | Сводный отчёт за период | **Реализовано:** `pdfkit`, задача `{"type":"report","reportDays":1}`. Файл: `data/reports/report-YYYY-MM-DD-last1d.pdf`. |
| **Telegram** | Отправка отчёта раз в сутки + по команде | **Реализовано:** раз в день в 9:00 (по умолчанию 09:00 UTC, настраивается `CRON_REPORT_SCHEDULE`) и **по запросу** — команда /report или «отчёт» в боте. PDF уходит в чаты из `TELEGRAM_REPORT_CHAT_IDS` или в чат, откуда пришёл запрос. |

## Этапы реализации

1. **Поиск и парсинг** — реализовано: при задаче `{"type":"collect"}` или тексте про сбор данных по безопасности в горнодобыче агент вызывает `searchMiningSafety()` (DuckDuckGo), сохраняет в SQLite `data/agent.db`. **Cron:** при `CRON_ENABLED=1` раз в сутки (по умолчанию 00:00 UTC, `CRON_COLLECT_SCHEDULE=0 0 * * *`) в очередь добавляется задача сбора.
2. **БД** — **реализовано:** SQLite в `data/agent.db`, таблица `articles` (url, title, snippet, search_query, fetched_at). Дубликаты по url не сохраняются. Модуль `src/db.js`: `insertArticles()`, `getArticlesSince(days)`, `getStats()`.
3. **Ежедневный отчёт** — cron в 00:00 UTC (или нужное время) создаёт задачу «сформировать отчёт за последние 24 ч»; модуль выбирает данные из БД, генерирует PDF, отправляет в Telegram.
4. **Telegram-бот** — команда типа `/report` или «пришли отчёт»: та же логика (сформировать PDF за последние N дней и отправить в чат).
5. **Интеграция с очередью** — типы задач в `payload`: `collect` (сбор), `report` (отчёт за период), `report_now` (отчёт по запросу и отправка в Telegram).

## Переменные окружения для сценария

- `OPENAI_API_KEY` — уже есть (для LLM-задач).
- `CRON_ENABLED` — `1` для включения ежедневного сбора (по умолчанию включён).
- `CRON_COLLECT_SCHEDULE` — расписание cron (UTC), по умолчанию `0 0 * * *` (каждый день в 00:00).
- `TELEGRAM_BOT_TOKEN` — для будущей отправки отчётов в Telegram.
- `TELEGRAM_REPORT_CHAT_IDS` — ID чатов для отчёта (через запятую).
- Для поиска: API-ключ не нужен (DuckDuckGo). На Railway: добавьте **Volume** с Mount Path `/app/data` — приложение использует `RAILWAY_VOLUME_MOUNT_PATH` и пишет БД на volume (см. [RAILWAY.md](../RAILWAY.md#4-volume-для-сохранения-бд-sqlite)).

## Как отправить задачу после настройки AGENT_URL

Когда в `.env` будут заданы `AGENT_URL` и при необходимости `WEBHOOK_SECRET`, из Cursor можно отправить задачу так:

```bash
npm run send-task "Тип: описание сценария мониторинга безопасности в горнодобыче (см. docs/mining-safety-monitor.md)"
```

Пока агент без доработок вернёт только текстовый ответ LLM. Реальное выполнение сценария — после реализации модулей выше.
