# Сценарий: мониторинг инноваций в горнодобывающей промышленности

Задача: круглосуточный сбор данных об инновациях в горнодобывающей отрасли → база данных материалов → ежедневный PDF-отчёт в Telegram (или по запросу).

## Что нужно добавить к текущему агенту

Текущий агент только передаёт текст в LLM и пишет ответ в логи. Для этого сценария нужны отдельные модули.

| Компонент | Назначение | Варианты реализации |
|-----------|------------|----------------------|
| **Поиск в интернете** | Сбор новостей, кейсов и технологических обзоров | **Реализовано:** DuckDuckGo через `duck-duck-scrape` (без API-ключа). Задача с `type: "collect"` или текст про «искать в интернете» + «иновации/инновации» + «горнодобыва» запускает поиск, результаты сохраняются в SQLite `data/agent.db`. |
| **Планировщик 24/7** | Запуск сбора по расписанию | **Реализовано:** `node-cron` в `src/index.js`. По умолчанию `CRON_COLLECT_SCHEDULE=0 * * * *` (каждый час, 24/7) → создаётся задача `collect`. |
| **База данных** | Хранение собранных материалов | SQLite (файл) или PostgreSQL (если есть на Railway); таблицы: источники, статьи, даты, теги |
| **Генерация PDF** | Сводный отчёт за период | **Реализовано:** `pdfkit`, задача `{"type":"report","reportDays":1}`. Формируется ежедневный отчёт за последние сутки. Файл: `data/reports/report-YYYY-MM-DD-last1d.pdf`. |
| **Telegram** | Отправка отчёта раз в сутки + по команде | **Реализовано:** раз в день в 9:00 (по умолчанию 09:00 UTC, настраивается `CRON_REPORT_SCHEDULE`) и **по запросу** — команда /report или «отчёт» в боте. PDF уходит в чаты из `TELEGRAM_REPORT_CHAT_IDS` или в чат, откуда пришёл запрос. |

## Этапы реализации

1. **Поиск и парсинг** — реализовано: при задаче `{"type":"collect"}` или тексте про сбор данных об инновациях в горнодобыче агент вызывает `searchMiningInnovation()` (DuckDuckGo), сохраняет материалы в SQLite `data/agent.db`. **Cron:** при `CRON_ENABLED=1` по умолчанию каждый час (`CRON_COLLECT_SCHEDULE=0 * * * *`) в очередь добавляется задача сбора.
2. **БД** — **реализовано:** SQLite в `data/agent.db`, таблица `articles` (url, title, snippet, search_query, fetched_at). Дубликаты по url не сохраняются. Модуль `src/db.js`: `insertArticles()`, `getArticlesSince(days)`, `getStats()`.
3. **Ежедневный отчёт** — cron раз в сутки (по умолчанию 09:00 UTC) создаёт задачу «сформировать отчёт за последние 24 ч»; модуль выбирает данные из БД, генерирует PDF, отправляет в Telegram.
4. **Telegram-бот** — команда типа `/report` или «пришли отчёт»: та же логика (сформировать PDF за последние N дней и отправить в чат).
5. **Интеграция с очередью** — типы задач в `payload`: `collect` (сбор), `report` (отчёт за период), `report_now` (отчёт по запросу и отправка в Telegram).

## Переменные окружения для сценария

- `OPENAI_API_KEY` — уже есть (для LLM-задач).
- `CRON_ENABLED` — `1` для включения планового сбора (по умолчанию включён).
- `CRON_COLLECT_SCHEDULE` — расписание cron (UTC), по умолчанию `0 * * * *` (каждый час, 24/7).
- `COLLECT_RESULTS_PER_QUERY` — сколько результатов брать на каждый поисковый запрос (по умолчанию 5).
- `TELEGRAM_BOT_TOKEN` — для будущей отправки отчётов в Telegram.
- `TELEGRAM_REPORT_CHAT_IDS` — ID чатов для отчёта (через запятую).
- Для поиска: API-ключ не нужен (DuckDuckGo). На Railway: добавьте **Volume** с Mount Path `/app/data` — приложение использует `RAILWAY_VOLUME_MOUNT_PATH` и пишет БД на volume (см. [RAILWAY.md](../RAILWAY.md#4-volume-для-сохранения-бд-sqlite)).

## Как отправить задачу после настройки AGENT_URL

Когда в `.env` будут заданы `AGENT_URL` и при необходимости `WEBHOOK_SECRET`, из Cursor можно отправить задачу так:

```bash
npm run send-task "Искать в интернете инновации в горнодобывающей промышленности"
```

Проверить результат можно через `GET /status` (поле `db`) и по PDF из `data/reports/`.
